## Q.1 Pipes

**Answer:**
Pipes in Angular are simple, powerful tools used to transform data directly in templates. They make template expressions cleaner, reusable, and more readable.

A Pipe takes in data, transforms it, and returns the modified result—similar to functions—but used inside HTML templates.
* ***Example:***

```html
        <span>{{ name | uppercase }}</span>
```
* **Uses**
    * Format data in templates
    * Avoid writing logic inside HTML
    * Reusable transformations
    * Improve readability


| Category       | Examples                                       |
| -------------- | ---------------------------------------------- |
| **Formatting** | `date`, `currency`, `percent`, `number`        |
| **String**     | `uppercase`, `lowercase`, `titlecase`, `slice` |
| **JSON**       | `json`                                         |
| **Custom**     | your own logic                                 |
| **Async**      | `async` pipe (handles Observables/Promises)    |

## Q.2 Pure Pipe vs Inpure Pipe

**Answer:**

* **Pure Pipe**
    * Executes only when input changes
    * Performs better
    * Suitable for pure functions, immutable inputs

```ts
@Pipe({ name: 'multiply', pure: true })
export class MultiplyPipe implements PipeTransform {
  transform(value: number, factor: number) {
    return value * factor;
  }
}
```

* **Impure Pipes**
    * Executes on every change detection cycle
    * Used when data is mutable (arrays, objects)
    * Can impact performance
```ts
@Pipe({ name: 'filterList', pure: false })
export class FilterListPipe implements PipeTransform {
  transform(list: any[], search: string) {
    return list.filter(item => item.includes(search));
  }
}
```

## Q.4 Angular Forms - Template vs Reactive

**Answer:**

1. **Template-driven Forms**
    * These forms are driven by the template (HTML).
    Most of the logic is in HTML using Angular directives.
    * **Best For:**
        * Simple forms
        * Small applications
        * Minimal logic
        * Two-way binding
```html
<form #myForm="ngForm" (ngSubmit)="submit(myForm)">
  <input name="username" ngModel required />
  <input name="email" ngModel email />
  <button type="submit">Submit</button>
</form>
```
```ts
submit(form: any) {
  console.log(form.value);
}
```

2. **Reactive Forms**
    * These forms are driven by TypeScript code.
    The form model is created and maintained in the component.
    * **Best For**
        * Complex, dynamic, large forms
        * Dynamic validation
        * High testability
        * Better performance
        * More control
```ts
form = new FormGroup({
  username: new FormControl('', Validators.required),
  email: new FormControl('', [Validators.required, Validators.email]),
});

submit() {
  console.log(this.form.value);
}
```
```html
<form [formGroup]="form" (ngSubmit)="submit()">
  <input formControlName="username" />
  <input formControlName="email" />
  <button>Submit</button>
</form>

```
3. 
| Feature           | Template-driven              | Reactive                       |
| ----------------- | ---------------------------- | ------------------------------ |
| **Form creation** | In HTML                      | In TypeScript                  |
| **Binding**       | Two-way `(ngModel)`          | One-way `[formControl]`        |
| **Data model**    | Generated by Angular         | Defined by developer           |
| **Validation**    | In template using attributes | In TS using Validators API     |
| **Form updates**  | Automatic                    | Explicit (setValue/patchValue) |
| **Unit testing**  | Hard                         | Very easy                      |
| **Scalability**   | Not good                     | Excellent                      |
| **Dynamic forms** | Hard                         | Very easy                      |
| **Performance**   | Lower                        | Higher (immutable)             |

## Q.6 Change Detection

**Answer:**

Change Detection is the mechanism Angular uses to check the component tree and update the DOM whenever an application's model (data) changes.

Whenever something changes → Angular updates the UI.

Angular checks the UI by running a change detection cycle through its component tree.

By default, Angular uses Zone.js to detect events such as:`clicks`, `HTTP responses`, `Timers (setTimeout)`, `Promises`, `Input changes`.
Whenever any of these occur → Angular triggers change detection automatically.

```pgsql
User Action / Event → Zone.js detects → NgZone triggers CD →
  Angular walks component tree (top → bottom) →
    Compares old value vs new value →
      Updates DOM if changed.

```
* **Two Change detection strategies**
    1. **Default (CheckAlways)**
        * Every component is checked on every CD cycle
        * Even if input hasn't changed
        * Good for correctness, not for performance
        ```ts
            changeDetection: ChangeDetectionStrategy.Default
        ```
    2. **OnPush (CheckOnce)**
        Angular checks the component only when:
        * Checks only affected components, skips unnecessary checks
        * Input reference changes
        * Event triggered inside the component
        * Observable emits (async pipe)
        * Manually triggered with `markForCheck()` or `detectChanges()`
        ```ts
            changeDetection: ChangeDetectionStrategy.OnPush
        ```
## Q.7 `@HostListener` and `@HostBinding`

**Answer:**
Both are Angular decorators used inside Directives or Components to interact with the host element (the element on which the directive/component is applied).

1. **@HostListener**
    * `@HostListener` lets you listen to events on the host element.
    * Used to capture events like : `click`, `mouseover`, `keypress`, `scroll`, etc.
    ```ts
        @Directive({
            selector: '[appClickHighlight]'
        })
        export class ClickHighlightDirective {

            @HostListener('click')

            onClick() {
                console.log('Host element clicked!');
            }
        }
    ```
    ```html
        <p appClickHighlight>Click me</p>
    ```

2. **@HostBinding**
    * `@HostBinding` allows you to bind a property of the host element—like class, style, attribute, etc.
    * Used to dynamically update host element DOM properties.
    ```ts
        @HostBinding('style.background') bgColor: string = 'yellow';
    ```

| Feature    | @HostListener            | @HostBinding             |
| ---------- | ------------------------ | ------------------------ |
| Purpose    | Listen to events on host | Bind properties of host  |
| Use        | Handle interactions      | Modify DOM / attributes  |
| Example    | click, scroll            | style, class, attributes |
| Applied on | Methods                  | Variables                |

```ts
@HostBinding('class.active') isActive = false;

@HostListener('click')
toggle() {
  this.isActive = !this.isActive;
}

```
## Q.8 View Encapsulation

**Answer:**
In Angular, View Encapsulation controls how styles defined in a component are applied to the DOM.
It determines whether your component’s CSS affects only its template or leaks outside, and whether global styles can affect your component.

* **Types of View Encapsulation**

1. **Emulated (Default)**
    * Angular simulates Shadow DOM behavior by scoping styles using unique attributes.
    * Styles apply only to the component, not globally.
    * Angular adds a unique attribute like `_ngcontent-a` to HTML elements.
    ```ts
        @Component({
            selector: 'app-test',
            templateUrl: './test.component.html',
            styleUrls: ['./test.component.css'],
            encapsulation: ViewEncapsulation.Emulated
        })
    ```
2. **ShadowDom**
    * Uses the browser’s native Shadow DOM for true encapsulation.
    * Styles are completely isolated—no leaking inside or outside.
    ```ts
        @Component({
            selector: 'app-test',
            templateUrl: './test.component.html',
            styleUrls: ['./test.component.css'],
            encapsulation: ViewEncapsulation.ShadowDom
        })
    ```
3. **None**
    * Styles are added to the global CSS.
    * Every component on the page can be affected.
    ```ts
        @Component({
            selector: 'app-test',
            templateUrl: './test.component.html',
            styleUrls: ['./test.component.css'],
            encapsulation: ViewEncapsulation.None
        })
    ```
    * Useful for UI libraries where styles need to be global.
    * Easy to override styles.

## Q.9 Performance Optimization Techniques of Angular application

**Answer:**

1. *Change Detection Optimization (Most Important)*
    * Use `ChangeDetectionStrategy.OnPush`
    * Only re-renders when:
        * Input reference changes
        * Event occurs inside component
        * Observable emits (with async pipe)
    * Huge performance gain in large apps

2. *Use `trackBy` in `*ngFor`*
    * trackBy renders only changed items, not entire list.
    * Prevents DOM re-creation
    * Faster list updates

3. *Lazy Loading Modules*
    * Load only required modules instead of full app.
    * Reduces initial bundle size
    * Improves load time

4. *Use `Async Pipe` Instead of Manual Subscription* 
    * Auto unsubscribe
    * Prevents memory leaks
    * Triggers OnPush correctly

5. *Use AOT (Ahead-of-Time Compilation)*
    * Compiles templates at build time
    * Faster rendering
    * Smaller bundle size

6. *Optimize Images & Assets*
    * Use lazy loading images
        ```HTML
            <img src="image.jpg" loading="lazy" />
        ```
    * Compress images
    * Use WebP format

7. *Use Signals (Latest Angular 🔥)*
    * Fine-grained reactivity
    * Replaces heavy change detection cycles

8. *Avoid Memory Leaks*
    * Always unsubscribe:
        ```JS
            ngOnDestroy() {
                this.subscription.unsubscribe();
            }
        ```

## Q.10 AOT vs JIT

**Answer:**

* **JIT (Just-In Time):**
    * Compilation happens in the browser at runtime
    * Angular compiles templates when the app runs
    * **Flow:**
        ```
            TS + HTML → (browser compiles) → JavaScript → Run app
        ```
    * **Example:** (Default during development)
        ```bash
            ng serve
        ```
    * *Pros:*
        * Faster build time
        * Good for development
        * Easier debugging
    * *Cons:*
        * Slower app startup
        * Larger bundle size
        * Templates compiled in browser → less secure

* **AOT (Ahead of Time):**
    * Compilation happens during build time.
    * Templates are converted to JavaScript before app runs.
    * **Flow:**
        ```
            TS + HTML → (build time compile) → JS → Browser runs directly
        ```
    * **Example:**
        ```bash
            ng build --configuration production
        ```
    * *Pros:*
        * Faster Rendering
            No compilation in browser
        * Smaller Bundle Size
            Removes Angular compiler from bundle
        * Better Security
            No runtime template parsing → prevents injection attacks
        * Early Error Detection
            Template errors caught during build


## Q.11 `trackBy` in Angular (`*ngFor`)

**Answer:** 
    `trackBy` is used to optimize list rendering in Angular by telling Angular how to uniquely identify items in a list.
* It helps Angular:
    * Identify which items actually changed
    * Reuse existing DOM elements
    * Avoid unnecessary re-rendering
```html
    <li *ngFor="let item of items; trackBy: trackById">
        {{ item.name }}
    </li>
```
```ts
    trackById(index: number, item: any) {
        return item.id;
    }
```
* *Without trackBy*
    * Angular compares objects by reference
    * Even same data → treated as new → full DOM re-render ❌
* *With trackBy*
    * Angular compares using unique ID
    * Only changed items update ✅

* **When to Use `trackBy`**
    * Large lists
    * Frequently updating data
    * API refresh scenarios
    * Tables, dropdowns, chat apps

* **Advanced Example:**
    ```html
        <div *ngFor="let user of users; trackBy: trackByUser">
            {{ user.name }}
        </div>
    ```
    ```ts
        //Good
        trackByUser(index: number, user: any) {
            return user.id || user.email;
        }

        //Bad Practice
        /*
        trackByUser(index: number, user: any) {
            return user.id || user.email;
        }
        */

    ```
    * Always use unique identifier
    * Avoid using index (bad practice)
        * Because index changes when list order changes

## Q.13 Promises

**Answer:** A Promise is an object that represents the result of an asynchronous operation (like API calls, file reading, timers).

* A Promise has 3 states:
    1. Pending → Initial state
    2. Fulfilled (Resolved) → Success
    3. Rejected → Failure

* 
```js

//creating a promise
const promise = new Promise((resolve, reject) => {
  let success = true;

  if (success) {
    resolve("Operation Successful");
  } else {
    reject("Operation Failed");
  }
});

//Consuming Promise
promise
  .then(result => {
    console.log(result);
  })
  .catch(error => {
    console.log(error);
  });
```
* **Example (API Simulation):**

```js
function fetchData() {
  return fetch('https://jsonplaceholder.typicode.com/posts/1')
    .then(response => {
      if (!response.ok) {
        throw new Error('Network response was not ok');
      }
      return response.json();
    });
}

fetchData()
  .then(data => console.log("Data received:", data))
  .catch(err => console.error("Error:", err));
```

* **Promise Chaining:**
    * Avoids nested callbacks
    * Clean flow of async operations
    * 
    ```js
    getUser()
        .then(user => getOrders(user.id))
        .then(orders => getOrderDetails(orders))
        .then(details => console.log(details))
        .catch(err => console.log(err));
        .finally(() => console.log("Done"));
    ```
## Q.14 Promise API Methods

**Answer:**

1. `Promise.all()`
    * Runs multiple promises in parallel
    * Fails if any one fails
    * 
    ```js
    Promise.all([p1, p2, p3])
        .then(results => console.log(results));
    ```

2. `Promise.allSettled()`
    * Returns all results (success + failure)
    * 
    ```js
    Promise.allSettled([p1, p2])
        .then(results => console.log(results));
    ```
3. `Promise.race()`
    * Returns first settled (completed) promise (either success or failure)
    * 
    ```js 
    Promise.race([p1, p2])
        .then(result => console.log(result));
    ```
4. `Promise.any()`
    * Returns first fullfilled (successful) promise
    * 
    ```js
        Promise.any([p1, p2])
            .then(result => console.log(result));
    ```

* **Note: `fetch()`**

    * fetch() is a built-in browser API used to make HTTP requests (GET, POST, PUT, DELETE, etc.). It returns a Promise, making it easy to handle async operations.
    * When you call fetch(), the response you get is NOT actual data yet.
        It returns a Response object, not parsed JSON.
        So  `.json()` is Needed

        * Reads the stream
        * Converts it into JavaScript object (JSON)
        * Returns a Promise
    * We can use other Body Parsers as well 

        *   | Method           | Use Case                    |
            | ---------------- | --------------------------- |
            | `res.json()`     | JSON data                   |
            | `res.text()`     | Plain text                  |
            | `res.blob()`     | Binary data (files, images) |
            | `res.formData()` | Form data                   |

    * 
    ```js
        //exaple asyn await 
        const response = await fetch('https://jsonplaceholder.typicode.com/posts/1');
        const data = await response.json(); //Since fetch returns promise 

        //example promise chaining

        fetch('https://jsonplaceholder.typicode.com/posts/1')
            .then(res => {
                if (!res.ok) {
                throw new Error('Request failed');
                }
                return res.json();
            })
            .then(data => console.log(data))
            .catch(err => console.error("Error:", err));
    ```
    * Returns ReadableStream, so use:
        1. response.json()
        2. response.text()
    
* **Example:**
```js
function fetchPost() {
  return fetch('https://jsonplaceholder.typicode.com/posts/1')
    .then(res => res.json());
}

function fetchUser() {
  return fetch('https://jsonplaceholder.typicode.com/users/1')
    .then(res => res.json());
}

function fetchComments() {
  return fetch('https://jsonplaceholder.typicode.com/comments/1')
    .then(res => res.json());
}

// Run all promises together
Promise.all([fetchPost(), fetchUser(), fetchComments()])
  .then(results => {
    const [post, user, comment] = results;

    console.log("Post:", post);
    console.log("User:", user);
    console.log("Comment:", comment);
  })
  .catch(error => {
    console.error("One of the APIs failed:", error);
  });
```

## Q.18 Hot and Cold Observables

**Answer:**
1. **Cold Observable:**
    * Starts producing values only when you subscribe
    * Each subscriber gets its own execution
    * Data source is created inside the observable
    * 
    ```js
    import { Observable } from 'rxjs';

    const cold$ = new Observable(observer => {
    console.log('Executing observable...');
    observer.next(Math.random());
    });

    cold$.subscribe(val => console.log('Subscriber 1:', val));
    cold$.subscribe(val => console.log('Subscriber 2:', val));
    ```
    * *Output:*
    ```
    Executing observable...
    Subscriber 1: 0.23

    Executing observable...
    Subscriber 2: 0.78
    ```
    * Each subscription runs separately
    * Different values for each subscriber
    * *Example:*
        ```js
        this.http.get('/api/data').subscribe();
        this.http.get('/api/data').subscribe();
        ```
    * Two subscriptions → 2 API calls
    * HTTP Observables are Cold by default

2. **Hot Observable:**
    * Starts emitting values regardless of subscription
    * Shared among all subscribers
    * Subscribers get same data stream
    * 
    ```js
    import { Subject } from 'rxjs';

    const hot$ = new Subject();

    hot$.subscribe(val => console.log('Subscriber 1:', val));
    hot$.subscribe(val => console.log('Subscriber 2:', val));

    hot$.next(Math.random());
    ```
    ```text
    Subscriber 1: 0.45
    Subscriber 2: 0.45
    ```
    * Same value shared 
    * One execution
    * *Example:*
    ```js
    fromEvent(button, 'click').subscribe(console.log);
    ```
    * Click happens regardless of subscription
    * All subscribers get same event

* **Convert Cold → Hot:** Using `share()` or `shareReplay()`

```js
import { shareReplay } from 'rxjs/operators';

const data$ = this.http.get('/api/data').pipe(
  shareReplay(1)
);

data$.subscribe();
data$.subscribe();
```
* Only one API call
* Shared result


## Q.19 Subjects 

**Answer:** A Subject is a special type of Observable that acts as both:

* 👉 Observable (you can subscribe)
* 👉 Observer (you can emit values using .next())

* Normal Observables:

    * ❄️ Cold → each subscription gets separate execution

* 👉 Subjects:

    * 🔥 Hot → shared execution across subscribers
    * Useful for event sharing, state updates

* **Example:**
    ```js
    import { Subject } from 'rxjs';

    const subject = new Subject();

    subject.subscribe(val => console.log('Subscriber 1:', val));
    subject.subscribe(val => console.log('Subscriber 2:', val));

    subject.next(1);
    subject.next(2);
    ```

    ```
    Subscriber 1: 1
    Subscriber 2: 1
    Subscriber 1: 2
    Subscriber 2: 2
    ```
    * Same data shared to all subscribers

* *Note*
    ```js
    const subject = new Subject();

    subject.next(1); // emitted before subscribe

    subject.subscribe(val => console.log(val));

    subject.next(2);
    ```
    * Output:
    ```
        2
    ```
* **`Behaviour Subject`: Most Used in Angular**
    * Requires initial value
    * Emits latest value immediately to new subscribers
    * 
    ```js
    import { BehaviorSubject } from 'rxjs';

    const subject = new BehaviorSubject(0);

    subject.subscribe(val => console.log('A:', val)); // gets 0

    subject.next(1);

    subject.subscribe(val => console.log('B:', val)); // gets latest (1)
    ```
    ```
        A: 0
        A: 1
        B: 1
    ```
    * Perfect for state management

* **`Replay Subject`:**
    * Stores previous values and replays them to new subscribers
    * 
    ```js
    import { ReplaySubject } from 'rxjs';

    const subject = new ReplaySubject(2); // buffer size = 2

    subject.next(1);
    subject.next(2);
    subject.next(3);

    subject.subscribe(val => console.log(val));
    ```
    * **Output:**
    ```
    2
    3
    ```
    * Useful for caching

* **`Async Subject`:**
    * Emits only last value when completed
    * 
    ```js
    import { AsyncSubject } from 'rxjs';

    const subject = new AsyncSubject();

    subject.subscribe(val => console.log(val));

    subject.next(1);
    subject.next(2);
    subject.complete();

    //Output
    2
    ```
    * Used when only final result matters

* **Angular Example (Service Communication)**
    * Sharing Data between two sibling components (not parent->child) component A -> component B
    ```js
    // service.ts
    import { BehaviorSubject } from 'rxjs';

    @Injectable({ providedIn: 'root' })
    export class DataService {
    private dataSubject = new BehaviorSubject(null);
    data$ = this.dataSubject.asObservable(); //Always expose as observable

    updateData(data: any) {
        this.dataSubject.next(data);
    }
    }
    ```
    ```TS
    //component A
    this.dataService.updateData("Hello from A");
    ```

    ```TS
    //component B
    this.dataService.data$.subscribe(data => {
        console.log(data);
    });
    ```

## Q.20 Signals

**Answer:**
Signals are a modern reactive primitive in Angular (introduced in Angular 16+) that enable fine-grained reactivity without relying heavily on Zone.js or full change detection cycles.
👉 Think of signals as reactive variables that automatically notify Angular when they change.

* *Signals:* A signal is a wrapper around a value that notifies interested consumers when that value changes. They can hold any type of data and are read by calling them as functions (e.g., count()). This function call allows Angular to track the dependency.

* **Fine-Grained Updates:** Angular knows exactly which components, computed values, or effects depend on the changed signal. Instead of re-checking the entire component tree (as in traditional zone-based change detection), Angular only re-runs the specific logic or updates the exact parts of the UI that are affected. This process is efficient because it avoids unnecessary work.

* **Dependency Tracking:** When a signal's getter function is called within a "reactive context" (like a component template, computed(), or effect()), Angular records the consuming context as a dependent of that signal.

* **Change Notification:** When a writable signal's value is changed using .set() or .update(), it marks its dependent consumers as "dirty" or "stale".

* **Types of Signals**
    1. **Writable Signals:** Created with the `signal()` function, these can be updated using the `.set()` method (replaces the entire value) or the `.update()` method (computes a new value based on the current one).

        ```ts
        import { signal } from '@angular/core';

        count = signal(0);

        // Read
        console.log(this.count());

        // Update
        this.count.set(5);

        // Update based on previous value
        this.count.update(val => val + 1);
        ```
        ```html
        <p>{{ count() }}</p>
        <button (click)="count.set(count() + 1)">Increment</button>
        ```
    2. **Computed Signals:** Created with the computed() function, these derive their value from one or more other signals. They are read-only and automatically recalculate their value when their dependencies change. They are lazily evaluated and memoized, meaning the calculation only runs when the value is read and the result is cached until a dependency changes.
        ```ts
        import { signal, computed } from '@angular/core';

        price = signal(100);
        tax = signal(10);

        total = computed(() => this.price() + this.tax());
        ```
        ```html
        <p>Total: {{ total() }}</p>
        ```
        * Automatically recalculates when dependencies change

    3. **Effects:** Created with the effect() function, these are operations that run whenever any of the signals they read change. Effects are used for side effects, such as logging data, manually updating the DOM, or syncing with localStorage. They run at least once and are automatically cleaned up when the associated component or service is destroyed.
        ```ts
        import { effect } from '@angular/core';

        effect(() => {
            console.log("Count changed:", this.count());
        });
        ```
        * Automatically runs when `count` updates
        * 
    4. **Readonly Signal**
        * Prevents external modification
        ```ts
        readonlyCount = this.count.asReadonly();
        ```
* **Signals + API (Hybrid Approach)**
    ```ts
    data = signal([]);

    ngOnInit() {
    this.http.get('/api/data').subscribe(res => {
        this.data.set(res);
    });
    }
    ```
    * Use Observables for API
    * Signals for UI state

* **Advanced Example (Form)**
    ```ts
    name = signal('');
    age = signal(0);

    isValid = computed(() => {
    return this.name().length > 0 && this.age() > 18;
    });
    ```
    ```html
    <input (input)="name.set($event.target.value)">
    <input type="number" (input)="age.set($event.target.value)">

    <p *ngIf="isValid()">Valid User</p>
    ```

* **Convert Observable → Signal**
    * Angular provides `toSignal()`
    ```ts
    import { toSignal } from '@angular/core/rxjs-interop';

    data$ = this.http.get('/api/data');

    //dataSignal = toSignal(this.data$, { initialValue: [] });

    //With Transformation
    dataSignal = toSignal(
        this.http.get('/api/data').pipe(
            map((res: any) => res.items)
        ),
        { initialValue: [] }
    );
    ```
    ```html
    <div *ngFor="let item of dataSignal()">
        {{ item.name }}
    </div>
    ```
* **Using Signals to Share Data Between Components:**
    1. **Create Shared Service**
    ```ts
    import { Injectable, signal } from '@angular/core';

    @Injectable({ providedIn: 'root' })
    export class DataService {

        // shared signal
        private messageSignal = signal<string>('Initial Message');

        // expose readonly (best practice)
        message$ = this.messageSignal.asReadonly();

        // update method
        updateMessage(newMessage: string) {
            this.messageSignal.set(newMessage);
        }
    }
    ```
    2. **Component A (Sender):**
        ```ts
        import { Component } from '@angular/core';
        import { DataService } from './data.service';

        @Component({
        selector: 'app-sender',
        template: `
            <input #msg type="text">
            <button (click)="send(msg.value)">Send</button>
        `
        })
        export class SenderComponent {

            constructor(private dataService: DataService) {}

            send(value: string) {
                this.dataService.updateMessage(value);
            }
        }
        ```
    3. **Component B (Receiver):**
        ```ts
        import { Component } from '@angular/core';
        import { DataService } from './data.service';

        @Component({
        selector: 'app-receiver',
        template: `
            <h2>Message: {{ message() }}</h2>
        `
        })
        export class ReceiverComponent {

            message = this.dataService.message$;

            constructor(private dataService: DataService) {}
        }
        ```

A Singleton Service means:
👉 Only one instance of a service exists throughout the entire application

"A Singleton Service in Angular is a service that has only one instance throughout the application. It is typically created using `providedIn: 'root'`, ensuring that all components share the same instance via Angular's dependency injection system."

* **Advantages of Singleton service**
    * Shared data across components
    * Maintain global state
    * Avoid multiple instances (memory efficient)
    * Centralized logic (API, auth, config)

* `providedIn: 'root'` → registers service in root injector
    👉 Root injector = one instance for entire app

* *Scenario:*
    * Lets say you have `providedIn: 'root'` in service as well as `providers: [UserService]` in component.

    ```ts
    //Service
    @Injectable({
        providedIn: 'root'
    })
    export class UserService {}

    //Component
    @Component({
        standalone: true,
        providers: [UserService]
    })
    export class SomeComponent {}
    ```
    * Two separate instances are created
        1. `providers: [UserService]` in component → creates new local instance
        2. `providedIn: 'root'` → creates global instance

    * **🔥Note:** Component-level providers override root
        ```ts
            @Injectable({ providedIn: 'root' })
            export class CounterService {
                count = 0;
            }
        ```
        ```ts
        //component A using Local instance by using providers

        @Component({
            standalone: true,
            providers: [CounterService],
            template: `<button (click)="inc()">Increment</button>`
        })
        export class AComponent {
            constructor(public cs: CounterService) {}
                inc() {
                    this.cs.count++;
                    console.log('A:', this.cs.count);
                }
        }
        ```
        ```ts
        //component B using root instance
        
        @Component({
            standalone: true,
            template: `<p>{{ cs.count }}</p>`
        })
        export class BComponent {
            constructor(public cs: CounterService) {}
        }
        ```
        *   Component A → uses local instance
            Component B → uses root instance


## Q.22 Interceptors

**Answer:**
An Interceptor is a service that lets you intercept and modify HTTP requests and responses globally.

* **Uses:**
    * Add auth token to every request
    * Handle errors globally
    * Log requests/responses
    * Show loader/spinner
    * Modify headers or body

* **Implement Interceptors:**
```ts
import { Injectable } from '@angular/core';
import {
  HttpInterceptor,
  HttpRequest,
  HttpHandler,
  HttpEvent
} from '@angular/common/http';
import { Observable } from 'rxjs';

@Injectable()
export class AuthInterceptor implements HttpInterceptor {

  intercept(req: HttpRequest<any>, next: HttpHandler): Observable<HttpEvent<any>> {
        const token = 'my-secret-token';

        const clonedReq = req.clone({
        setHeaders: {
            Authorization: `Bearer ${token}`
        }
        });

        return next.handle(req).pipe(
            catchError(error => {
                console.error('Error:', error);
                return throwError(() => error);
            })
        );
    }
}
```

* **Register Interceptor (Standalone)**
```ts
import { provideHttpClient, withInterceptors } from '@angular/common/http';

bootstrapApplication(AppComponent, {
  providers: [
    provideHttpClient(
      withInterceptors([authInterceptorFn])
    )
  ]
});
```
* **Functional Interceptor (Latest Angular 🔥):**
```ts
import { HttpInterceptorFn } from '@angular/common/http';

export const authInterceptorFn: HttpInterceptorFn = (req, next) => {
  const cloned = req.clone({
    setHeaders: { Authorization: 'Bearer token' }
  });

  return next(cloned);
};
```

## Q.42 `Debouncing` and `Throttling`

**Answer:**
1. **`Debouncing`:** Debouncing ensures a function runs only after a set time has passed since the last call (ideal for search inputs).
    * **How it works:** When an event is triggered, a timer starts. If the event is triggered again before the timer ends, the timer resets. The function executes only after the timer finishes without being reset.
    * **Use Cases:**
        * Search box autocomplete (API calls only after user stops typing).
        * Window resizing (re-calculating layout only after resizing stops).
        * Form validation on input.
    * **Example:**
    ```js
    fromEvent(inputElement, 'keyup')
            .pipe(
                debounceTime(300)
            )
        .subscribe(() => console.log('API call'));
    ```
2. **`Throttling`:** Throttling ensures a function runs at most once every set interval (best for scrolling/resizing)
    * **How it works:** Guarantees that a function runs regularly at a fixed interval (e.g., once every 1000), regardless of how many times the event fires.
    * **Use Cases:**
        * `scroll` event handlers (updating UI elements while scrolling).
        * `mousemove` event tracking.
        * Game mechanics (limiting fire rate of a weapon).
    * **Example:**
    ```js
    fromEvent(window, 'scroll')
            .pipe(
                throttleTime(1000)
            )
        .subscribe(() => console.log('Scroll event'));
    ```
## Q.43 Event Propagation vs Event Delegation vs Event Capturing vs Event Bubbling

**Answer:**
1. **Event Propagation:** Event propagation is the process of how events travel through the DOM in three phases: capturing, target, and bubbling
    * 
    ```html
    <div id="parent">
        <button id="child">Click Me</button>
    </div>
    ```
2. **Event Capturing:** Event goes from parent → child (top -> bottom)
    * 
    ```js
    document.getElementById('parent').addEventListener(
        'click',
        () => console.log('Parent clicked'),
        true // enable capturing
    );

    document.getElementById('child').addEventListener(
        'click',
        () => console.log('Child clicked'),
        true
    );

    // Output
    Parent clicked
    Child clicked
    ```

3. **Event Bubbling:** Event goes from child → parent → document (bottom -> top)
    * 
    ```js
    document.getElementById('parent').addEventListener('click', () => {
        console.log('Parent clicked');
    });

    document.getElementById('child').addEventListener('click', () => {
        console.log('Child clicked');
    });
    
    //Output
    Child clicked
    Parent clicked
    ```

4. **Event Delegation:** Event delegation is a technique where a parent handles events for its children, improving performance and handling dynamic elements efficiently.
    * Attach event listener to parent instead of multiple children
    * 
    ```js
    document.getElementById('parent').addEventListener('click', (e) => {
        if (e.target.tagName === 'BUTTON') {
            console.log('Button clicked:', e.target.id);
        }
    });
    ```
